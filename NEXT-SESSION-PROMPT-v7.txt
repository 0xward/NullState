## What THIS session verified (real Playwright, real screenshots)
v6 confirmed Playwright/Chromium launches in-sandbox but had NOT yet
built the standalone harness or looked at a real render. This session
did both:

1. Built `public/test.html` — loads `assets.js`+`entities.js` via plain
   `<script>` tags (no Next.js/bundler), constructs a real `Player`,
   exposes `window.__renderFrame(state)` to drive idle/walk/attack
   states from outside, and draws to a `<canvas>`. Served with
   `python3 -m http.server` from `public/`, driven with Playwright,
   screenshotted, and the images were actually viewed with the `view`
   tool — not just "should work," actually looked at pixel-for-pixel.
   **Keep this file for future QA, but delete it (or move it outside
   `public/`) before any real production deploy** — it's a dev-only
   harness, not something that should ship to users.

2. **Found and FIXED a direction bug.** The old code:
   ```js
   const lx=this._lastMove&&Math.abs(this._lastMove.x)>0.2?this._lastMove.x:this.facing;
   const ly=this._lastMove&&Math.abs(this._lastMove.y)>0.2?this._lastMove.y:0;
   ```
   fell back to `this.facing` (a left/right-only flag, defaults to 1,
   NEVER becomes 0) whenever `|_lastMove.x| <= 0.2`. That meant **pure
   vertical movement (walking straight up or straight down) always
   substituted facing=1 for lx**, and `lpcDirFromVec`'s `>=` tie-break
   then always resolved to horizontal. Confirmed via harness: a fresh
   player given ONLY `{down:true}` input rendered `_lpcDir=3` (right),
   not `2` (down) — screenshot showed the right-facing walk row instead
   of a down-facing pose. **This directly broke the "full 4-directional"
   decision v6 locked in — up/down never actually worked.**
   Fixed in `entities.js` (`Player.update()`) by just using
   `this._lastMove.x/.y` directly with no facing fallback — `_lastMove`
   already holds the correct persisted last-movement vector (only
   updates while actually moving, so it's still valid during idle
   frames too). Re-verified after the fix: fresh down-only → dir 2,
   fresh up-only → dir 0, then-right → dir 3. All correct now.

3. **NOT fixed, flagged for you to decide:** `Player.hitZone()` has the
   exact same `facing`-fallback pattern:
   ```js
   const lx=this._lastMove&&Math.abs(this._lastMove.x)>0.2?this._lastMove.x:this.facing;
   const ly=this._lastMove&&Math.abs(this._lastMove.y)>0.2?this._lastMove.y:0;
   ```
   This determines where the melee hit-zone lands relative to the
   player. If the same bug applies here (untested this session —
   didn't want to touch live combat hit-detection as a side effect of
   a visual-layering task without being asked), attacking while walking
   straight up/down could register the hit to the side instead of
   up/down. This is a **pre-existing combat-code bug, unrelated to LPC
   visuals** — decide whether to fix it the same way (just drop the
   `facing` fallback) or leave it, since it changes real gameplay hit
   registration, not just rendering. Quick to fix if you want it: same
   one-line change as the update() fix above.

4. **Recalibrated `LPC_HERO.scale`.** It was a placeholder `2.2`
   ("first estimate," never checked). Measured with PIL (not just
   frame size — actual opaque-pixel bounding box):
   - Old `knight_idle` frame: 29px-tall opaque content inside a 32px
     frame. At the old `scale:1.93`, that's `29*1.93 = 55.97px` on
     screen — confirms the code comment's "~56px" target is real.
   - LPC `BODY_male` walkcycle frame (down-facing row): 47px-tall
     opaque content inside its 64px frame.
   - Matching scale: `56/47 ≈ 1.19`, rounded to **1.2**. The old `2.2`
     was rendering the LPC hero at ~140px tall (measured via screenshot
     bbox: ~122px including anti-aliasing/shadow) — nearly 2.5x too big.
   - `foot:1.0` confirmed CORRECT as-is, no change needed: both sprites
     have their feet essentially flush with the frame's bottom edge
     (0px gap for old, 2px gap for LPC) — the ~56px is basically the
     entire drawn height, not shifted up (this was inferred to be
     safe in v6, now measured and confirmed).
   - Re-verified visually post-fix: idle frame now renders at a
     proportional size, and the `war_axe` slash overlay is still
     pixel-aligned to the body at the new scale (weapon/body/armor all
     share the same `dx/dy/dw/dh` math in `drawLPCComposite`, so scale
     changes apply uniformly — confirmed by screenshot, no separate
     weapon-anchor fix was needed).
