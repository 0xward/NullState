### Task 1 — Floor scaling wired in
**File:** `public/game-engine/entities.js`, `Enemy` constructor (~line 227).
Added a `window.NS_FLOOR_SCALE(depth)` call (defensive `|| {hpMul:1,dmgMul:1,xpMul:1}` guard,
matching the `window.NS_MONSTER_CONFIG||{}` style already used in `game.js`). The result multiplies
the existing `hpMul`/`dmgMul`/`xpMul` locals (which already carry elite/boss multipliers) — it
stacks, it does not replace.

**Design decision made (per the original task's "read carefully, don't guess" instruction):** the
pre-existing per-depth linear curves — `scale = 1+(depth-1)*0.14` (used only for `maxHp`), the
inline `(1+(depth-1)*0.12)` for `dmg`, and `(1+(depth-1)*0.1)` for `xp` — were left untouched and
independent. They represent each stat's own hand-tuned baseline growth curve (they were never
literally "the same scale" despite the misleading variable name `scale`; each stat had its own rate
already). The new floor-config layer (`+8%/floor` compounding, with fixed overrides at floors
5/10/15/20) is applied as an ADDITIONAL multiplicative layer on top of all of that. `rMul` (physical
size) was left alone entirely — it's driven only by elite/boss flags, not floor depth, and the task
didn't ask for monsters to visually grow every floor.

**Verified so far (Node, not browser):** ran `monster-config.js` in isolation and confirmed
`getFloorScale(f)` returns the right numbers — floor 1 → `{1,1,1}`, floor 5 → the Boss-Floor-I
overrides `{2.2, 1.8, 2.5}`, floor 6 → back to compounding `~1.469`, floor 10/20 → the bigger boss
overrides. Math checks out. **Not yet verified:** actually loading the game in a browser and
confirming floor 5/10/15/20 enemies visibly hit much harder/tankier than floor 4/6/etc, and that
normal enemy stats aren't now absurdly high by floor 20 (13x HP might be too much combined with the
pre-existing 0.14/floor curve — check `maxHp` numbers at floor 20 aren't fully unkillable; if they
are, that's a balance call for the project owner, not something to silently "fix" — flag it instead).

### Task 2 — `ranged` weapon behavior (Hunter's Bow) implemented
**Files:** `public/game-engine/game.js` (new `spawnProjectile()`, `updateProjectiles()`,
`drawProjectiles()`, plus `G.projectiles` added to game state init in `newGame()` and cleared on
floor change in `descend()` alongside `G.particles`/`G.dmgNums`), `hitTest()` restructured to branch
on `beh==='ranged'`.

**How it works:** when Hunter's Bow is equipped and the player attacks, `hitTest()` calls
`spawnProjectile(p, dmg)` instead of running the melee AoE loop at all — no phantom melee hit, and
(as a side effect of the branch) no decor-smashing sweep either for this weapon specifically. The
arrow travels in `p._lastMove` direction at 640px/s, max range 560px, drawn with plain canvas
primitives (rotated rect shaft + triangle arrowhead + two fletching marks — no new sprite files).
On first enemy touched it calls the new shared `applyHitToEnemy()` helper (extracted from the old
inline melee hit code so melee and ranged hits get IDENTICAL hit-stop/particle/knockback/shake
treatment — see refactor note below) and despawns. Arrows also despawn on hitting a wall
(`dun.isWall`) or exceeding max range.

**Refactor note:** `applyHitToEnemy(e, dmg, fromX, fromY, beh)` is now the single source of truth
for "deal damage + hit-stop + particles + knockback + shake + onEnemyKilled" — both the melee loop
in `hitTest()` and `updateProjectiles()` call it. This was a deliberate, scoped extraction (not a
big refactor) to satisfy the original task's "reuse existing helpers so it feels consistent"
requirement without duplicating ~20 lines of FX glue code twice. Double-check this didn't
subtly change melee behavior — the logic should be byte-for-byte equivalent to before, just moved
into a function.

**Not yet verified live:** equip Hunter's Bow, attack at range, confirm the arrow visibly travels,
connects, deals `p.atkDmg`-ish damage, and that other weapons are completely unaffected (still do
normal melee). Also confirm arrows don't visually clip through walls before `isWall` catches them
(640px/s at 60fps = ~10.6px/frame, should be fine but eyeball it).

### Task 3 — `slow` weapon behavior (Frost Spear) implemented
**Files:** `game.js` (`hitTest()`'s melee branch — Frost Spear's `slow` case now sets
`e._slowMul = 1 - slowPct` and `e._slowT = 3.0` seconds on hit, reading `slowPct` from the equipped
weapon's config, defaulting to 0.5 if missing), `entities.js` (`Enemy.update()` ticks `_slowT` down
and applies `_slowMul` to both the chase-speed calc and the walk-back-to-post calc;
`Enemy.draw()` adds a light-blue tint overlay (`source-atop` composite, tinting only the sprite's
own drawn pixels) plus a pulsing icy ring while `_slowT>0`).

**Design decision:** refreshing the timer on repeated hits (not stacking multiplier) was implemented
exactly as specced — a second Frost Spear hit resets `_slowT` back to 3.0 at the same `1-slowPct`
strength, it does not compound to make the enemy slower than `slowPct`.

**Not yet verified live:** equip Frost Spear, hit an enemy, confirm it visibly moves slower with the
blue tint/ring, and returns to normal speed + normal color after ~3s.
