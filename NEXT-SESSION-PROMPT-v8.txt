2. **Fixed `drawPreview()`/added `drawLPCPreview()` in `game.js`** —
   this was v7's item #3 ("still gear-unaware, refactor to call
   drawLPCComposite()"). Knight's char-select thumbnail now composites
   through the SAME `drawLPCComposite()` used by in-dungeon
   `Player.draw()`, instead of drawing the old Pixel Crawler
   `knight_idle` sprite. rogue/wizzard are UNCHANGED (still the old
   alpha-bbox-normalize path — they have no LPC sheets).
   **Caught a real bug via measurement, not just visual inspection:**
   my first version of `drawLPCPreview()`'s `cy` math was
   `(BOX-6)+dh*hero.foot` — but `drawLPCComposite()` already subtracts
   `dh*foot` internally to get the sprite's top edge from `cy`, so this
   was double-applying the foot offset and pushed the entire sprite
   below the visible 88px preview box. Confirmed via a Playwright
   `getImageData()` pixel scan of the actual canvas (not the DOM
   screenshot, which can look deceptively fine if a red debug outline
   is the only visible thing) — `found: false`, completely empty
   canvas. Fixed by using `cy = BOX-6` directly (the desired feet
   baseline) and letting `drawLPCComposite()` handle the foot
   subtraction itself. Re-verified after the fix: alpha bbox height on
   the actual canvas came back **74px** — exactly `TARGET`, confirmed
   correct. Screenshot also visually confirmed: down-facing LPC knight,
   correctly centered and sized in the thumbnail box.
   **Takeaway for next session: when verifying a canvas render, read
   the canvas's own pixel data via `getImageData()` in-page, don't just
   screenshot the DOM — a screenshot of an empty canvas inside a
   bordered/styled div can still "look like something's there" if you
   don't check the alpha channel directly.**

3. **Fixed `Player.hitZone()`'s facing-fallback bug** — this was v7's
   item #9, explicitly left for this session's judgment since it's
   combat logic, not rendering. Applied the same one-line-pattern fix
   as v7's `update()` fix: dropped the `this.facing` fallback, use
   `this._lastMove.x/.y` directly. Verified via the `test.html`
   harness: gave a fresh player `_lastMove={x:0,y:1}` (straight down),
   ran `startAttack()` + advanced into the strike window, called
   `hitZone()`. Before this session's fix this would have returned a
   hit zone offset to the side (`x: player.x+38`); after the fix it
   correctly returns `{x: player.x, y: player.y+30}` — pure downward
   offset, no horizontal drift. Melee hit registration on vertical
   attacks is now correct.
